# Contributing to Better Prompt

This covers the non-obvious parts of working on the plugin. For what it does, the pipeline, and settings, see the [README](README.md). For bug reports and pull requests, use the templates under `.github/` ‚Äî keep contributions concise and in your own words.

## Enable the git hook first if not enabled

```bash
git config core.hooksPath .githooks
```

A `post-commit` hook regenerates the changelog's Unreleased section and folds it into the commit that just finished. Fresh clones don't have `core.hooksPath` set, so run this once or the automation below silently does nothing.

## Commits drive the changelog

CHANGELOG.md is generated by [git-cliff](https://git-cliff.org) from **conventional commits** ‚Äî never hand-edit the top (Unreleased) section; it's regenerated.

- Type ‚Üí section: `feat`‚ÜíAdded, `fix`‚ÜíFixed, `perf`/`refactor`/`revert`/`docs`/`style`‚ÜíChanged.
- `chore:` (and `docs(changelog):`) are **excluded** from the changelog. Use `chore:` for build, config, template, and housekeeping work you don't want users to see.
- After each commit, the hook rewrites Unreleased and amends it into that same commit. It skips itself via `BP_CHANGELOG_HOOK=1` (set only by the release script ‚Äî don't set it for normal commits).
- The hook needs `git-cliff` on PATH; cutting a release also needs `jq`. Without `git-cliff`, the hook no-ops silently. You might need to install them.

## Cutting a release

`scripts/changelog.sh` manages the whole thing:

- `preview` ‚Äî print the regenerated Unreleased section.
- `unreleased` ‚Äî rewrite Unreleased in CHANGELOG.md.
- `release [x.y.z]` ‚Äî bump the version (auto-derived from commits if omitted), **sync `package.json` and `.claude-plugin/plugin.json`**, commit `chore(release): <version>`, and tag with bare semver (no `v`).

Then `git push --follow-tags` and create the GitHub release ‚Äî that triggers `.github/workflows/publish.yml`, which publishes to GitHub Packages and npmjs. Don't `npm publish` locally: the repo `.npmrc` scopes `@objctp` to GitHub Packages and would send the publish to the wrong registry.

## Agents and generated outputs

- `agents/*.md` (Claude Code frontmatter) is canonical. `npm run build` generates `.opencode/agents/` and the entire `dist/` package from it ‚Äî edit the source, rebuild, never the outputs.
- The `model:` field in `agents/*.md` (`haiku`|`sonnet`) only sets the agent's `temperature` in the OpenCode build (haiku‚Üí0.1, sonnet‚Üí0.3). Generated agents inherit the session model rather than pinning one; users override the actual model per stage via `better-prompt.local.md`.
- Adding or removing an agent or command? Update `.claude-plugin/plugin.json` too ‚Äî it's the manifest Claude Code loads.

## Tests, lint, format

- Shell tests run on **bashunit** under `tests/{unit,functional,acceptance}`. Assertions, spies, and mocks are documented in `.claude/rules/testing.md`; the `/shell-test-run` skill runs the suite with coverage.
- TS/JS is formatted with Biome (`biome.json`).
